# VM Detection

`ember --vm-detect <binary>` finds interpreter-style VM dispatchers — the
`movzx / advance-PC / jmp [table + idx*8]` shape that VM-protected code
uses for its outer run loop — and reports each VM's anatomy plus a
coarse classification of every handler in its table.

The detector is anatomy-only; it does not lift bytecode. It tells you
*which* function dispatches a VM, *where* the table lives, *which
register* holds the bytecode pointer, and *what kind of work* each
handler does. Phase 1c will turn those classifications into per-VM
opcode dictionaries; for now the output is enough to navigate a
protected binary by hand.

## The dispatcher pattern

Canonical shape, generated by every protector worth recognising:

```asm
movzx <idx>, byte ptr [<pc> + disp]    ; opcode <- *pc
add   <pc>, <pc_advance>                ; pc++   (optional; may live in handlers)
jmp   qword ptr [<table> + idx*8]       ; dispatch
```

Both the byte-load (`movzx ... byte/word ptr ...`) and the indirect
dispatch (`jmp` or `call qword ptr [<table> + idx*8]`) are required.
The PC advance is optional at the dispatcher — threaded VMs run the
`add pc, N` inside each handler instead.

The detector accepts two memory-operand shapes for the dispatch, gated
at `core/src/analysis/vm_detect.cpp:760-765`:

```cpp
const bool rip_form =
    m.base == Reg::Rip && m.has_disp &&
    m.scale == 8 && m.index != Reg::None;
const bool reg_form =
    m.base != Reg::None && m.base != Reg::Rip &&
    m.scale == 8 && m.index != Reg::None;
```

- **rip-form** — `[rip + disp + idx*8]`. The table VA is `rip_after +
  disp`. `has_disp` is required because rip-relative without a
  displacement isn't meaningful.
- **reg-form** — `[reg + idx*8]` where `reg` was set earlier by
  `lea reg, [rip+disp]` (or by VMProtect-class `call $+5; pop reg`,
  see below). `has_disp` is *not* required here: a clean
  `lea rsi, [rip+table]; jmp [rsi + rax*8]` leaves the displacement
  zero, which is the common case. Gating on `has_disp` would miss it.

When the gate fires, the index register's most-recent byte/word load is
checked; if it sits within `kReuseWindow = 16` real instructions
(`vm_detect.cpp:32`) the dispatch is recorded (`vm_detect.cpp:766-820`).
The walker captures into a `VmDispatcher`
(`core/include/ember/analysis/vm_detect.hpp:24`):

- `dispatch_addr`, `opcode_load_addr`, `function_addr` — site
  coordinates.
- `table_addr`, `table_entries`, `handlers` — the resolved table base,
  total entries walked before invalid, and the unique pointers within.
- `opcode_index_reg`, `opcode_size_bytes` — the un-canonicalised dst of
  the byte-load (so `eax` shows as `eax`, not `rax`) and 1 or 2 for
  byte/word.
- `pc_register`, `pc_disp`, `pc_advance` — the base register of the
  byte-load, its displacement, and any `inc / add / sub` of that
  register observed before the dispatch.
- `bytecode_addr` — the bytecode VA when the PC was set by a constant
  `lea pc, [rip+disp]` ahead of the loop. Zero means caller-supplied.

A dispatcher only emits when at least `kMinHandlerCount = 8`
(`vm_detect.cpp:26`) of its table entries point into executable code
(`vm_detect.cpp:801`). Stray indirect jumps through some C-style switch
table won't false-positive.

## Resilience

Real protectors (Tigress, VMProtect, Hyperion / Byfron, plus generic
flatteners) sprinkle nonsense between the byte-load and the dispatch
to break naive linear matchers. The detector layers four defenses.

### Junk-pad skip

`is_semantic_nop` (`vm_detect.cpp:89-186`) recognises the shapes
below. When a junk pad fires, the walker skips it without bumping
`age_counter` or touching the rolling register state, so an obfuscator
inflating the byte-load to dispatch gap doesn't blow `kReuseWindow`.

| Shape | Bytes (one example) | Source |
|-------|---------------------|--------|
| `nop` | `90` | universal |
| `xchg reg, reg` (same canonical reg) | `87 c0`, `48 87 c0` | VMProtect |
| `mov reg, reg` (same canonical reg) | `48 89 c9` | VMProtect / Tigress |
| `jmp $+next` | `eb 00` | Hyperion / Byfron flatteners |
| `lea reg, [reg]` / `lea reg, [reg+0]` | `48 8d 00` family | generic |
| `add/sub/or/xor reg, 0`, `and reg, -1` | `48 83 c0 00` etc. | VMProtect |
| `jcc $+next` (conditional, target = next insn) | `74 00` etc. | obfuscator pad |
| `cmp/test reg, reg` (same canonical reg) | `48 39 c0` etc. | paired with `jcc $+next` |

### Two-insn lookaheads

Some pads only register as a pair. The main loop handles two in-line:

- **`push reg; pop reg`** (`vm_detect.cpp:680-703`) — saves and
  restores the same register modulo flags / rsp side effects. Single-
  insn `is_semantic_nop` can't catch it; the loop peeks one
  instruction ahead and skips both when the popped register matches
  the pushed one.
- **`call $+5; pop reg`** (`vm_detect.cpp:704-743`) — VMProtect-class
  RIP capture. The call pushes the address of the pop; the pop
  materialises it into `reg`. Equivalent to `lea reg, [rip+0]` but
  without using `lea`, designed to defeat lea-only walkers. The pair
  is consumed and a synthetic `LeaRipRel` record is primed on the
  destination so reg-form dispatch resolution still works.

### CFG-aware walk

The walker isn't linear. A `visited` set guards against back-edges,
and direct unconditional `jmp imm` is followed when the target lands
inside an executable section (`vm_detect.cpp:932-939`). That's what
catches dispatchers split across blocks — `movzx` in block A, `jmp
[table + idx*8]` in block B reached via a forward `jmp imm`, with
`int3` padding sitting between them (a linear sweep would derail into
the padding).

Hard terminators stop the walk:

- `ret / ud2 / hlt / int3 / int` (`vm_detect.cpp:903-907`). `int3`
  matters because it's also the inter-function padding MSVC / gcc /
  clang emit; without it a linear walk from one function's last insn
  tumbles into the next function and mis-attributes its dispatcher.
- Any `call` (`vm_detect.cpp:914`) — caller-save regs (rax / rcx /
  rdx / r8..r11) get clobbered, which is exactly the dispatcher's
  working set.
- Indirect `jmp` (`vm_detect.cpp:920-922`) — control transfer with no
  fallthrough; the dispatch shape was already checked above.

Conditional `jmp` is **not** followed. The path divergence isn't
worth the rolling-state-merge complexity for a pattern that's almost
always linear at the dispatcher.

## Clustering — `VmInstance`

A threaded VM produces one `VmDispatcher` per handler, all pointing at
the same table. `group_vm_dispatchers` (`vm_detect.cpp:571-605`) folds
them into a `VmInstance` per unique `table_addr`. The classification
is per-site:

- A dispatcher's `function_addr` is **in** the cluster's `handlers`:
  **threaded site** (the handler ends with its own opcode-fetch +
  dispatch).
- A dispatcher's `function_addr` is **not** in `handlers`: **entry
  site** (a central run loop calls / falls through to handlers from
  outside the table).

A pure central VM has 1 entry site, 0 threaded sites. A pure threaded
VM has 0 entry sites and N threaded sites. Mixed VMs (some handlers
re-dispatch directly, others fall back to a central loop) carry
both — `vm_loop_split` and the threaded `ht_N` cluster in the fixture
exercise this.

Concrete example: an 8-handler threaded VM whose handlers each end
with `movzx eax, byte ptr [rdi]; inc rdi; jmp [rsi + rax*8]` produces
8 `VmDispatcher` records, all sharing one `table_addr`. They cluster
into one `VmInstance` with `entry_sites.size() == 0` and
`threaded_sites.size() == 8`. See VM #2 in the fixture golden.

## Phase 1b classification

`classify_vm_handler` (`vm_detect.cpp:299-569`) walks each handler
body and tallies which insn families it uses. Walks stop at the
caller-supplied `dispatch_addr` (for threaded handlers, so the
trailing dispatch shape isn't counted as body work), at `ret`, at an
unanchored indirect `jmp`, or at the first direct `call` (caller-save
regs gone past that point). A trailing-dispatch peek-ahead at
`vm_detect.cpp:347-377` also stops the body when a `movzx byte/word`
is followed within 3 real insns by an indirect `jmp` — that's the
threaded-tail shape and shouldn't count as Load.

Junk pads from `is_semantic_nop` are skipped here too, so a handler
whose only body is `nop / xchg rax, rax` reads as `Null`, not `Arith`.

### Precedence ladder

The kind selection at `vm_detect.cpp:485-491`:

```
Branch > Call > Store > Load > Arith > Return > Null
```

Body work wins over `Return` deliberately. A handler ending in
`add rax, rcx; ret` reads as `Arith add rcx`, not `Return`: the `ret`
is a function termination, the `add` is the meaningful op. Pure `ret`
handlers fall through to `Return`, which is what trivial vm_ret-style
opcodes look like in the wild.

`Branch` outranks `Call` and `Store` because conditional control flow
is the hardest thing to misclassify against — once you see a `jcc` /
`cmov` in a handler, you're almost certainly looking at a control-flow
opcode. `Store` outranks `Load` because a store implies a load too
(operand fetch + write), and the store is the salient action.

### Summary refinements

`HandlerClassification::summary` is a short kind-specific tag:

| Kind | Summary form | Example |
|------|--------------|---------|
| `Arith` | mnemonic + reg | `add rcx` |
| `Arith` (xor reg, reg, same canonical) | `clear` | `clear` |
| `Arith` (binary, imm rhs) | mnemonic + hex imm | `and 0xff` |
| `Load` | `[mem]` formatted | `[rcx]`, `[0x140002080]` |
| `Load` (base = `pc_register`) | `operand[+disp]` | `operand`, `operand+0x4` |
| `Store` | `[mem]` formatted | `[rcx]` |
| `Store` (base = `pc_register`) | `operand[+disp]` | `operand` |
| `Branch` | conditional mnemonic | `jne`, `cmovz` |
| `Call` (direct imm) | target VA | `0x14000132d` |

The `pc_register` refinement (`vm_detect.cpp:497-508`) is the
semantically interesting one: a load whose base is the VM's PC isn't
just reading some memory, it's fetching an *inline operand* from the
bytecode stream. Reporting it as `operand` rather than `[rdi]`
preserves that meaning — see VM #3 handler `[0x03]` in the golden,
which renders as `load operand` precisely because the `mov rax,
[rdi]` reads through the PC.

Rip-relative loads (`[rip+disp]`) are resolved to absolute VAs via
`format_mem_short` (`vm_detect.cpp:272-297`) so the user sees a
concrete address rather than a relative offset.

## Limits

Honest list of what the detector doesn't do yet:

- **Conditional jmp following.** Path divergence isn't merged, so a
  dispatcher whose byte-load and `jmp [table + idx*8]` are
  separated by a `jcc` (any flavour, including ones where both arms
  rejoin a few insns later) drops on the floor. Real VMs don't do
  this; obfuscators sometimes do.
- **Stack-slot junk pads.** `mov [rsp+X], reg; ...; mov reg,
  [rsp+X]` is a save/restore through the shadow-stack, equivalent to
  `push reg; pop reg` with arbitrary insns in between. The two-insn
  lookahead can't see it; we'd need a small shadow-stack tracker.
- **Multi-insn obfuscator patterns beyond what `is_semantic_nop`
  covers.** E.g. `lea reg, [reg + 1]; sub reg, 1` (net zero), or
  any arithmetic identity composed across two instructions. We catch
  the single-insn form (`add reg, 0`) but not the composed one.
- **Anchored RIP capture other than `call $+5; pop reg`.** Variants
  that push and pop with one or two filler insns between the call
  and the pop will miss; the detector requires the pair to be
  contiguous.
- **No cross-function arity / call-target reasoning.** A handler that
  jumps to a shared epilogue is classified by the first thing it
  does, even when the meaningful work happens at the shared block.

## The fixture

`tests/fixtures/make_vm_loop_pe.py` builds a tiny PE32+ with four
distinct VMs, plus the canonical golden in
`tests/golden/vm_loop_pe.vm_detect.txt`.

| VM | Source | What it tests |
|----|--------|---------------|

Handlers exercise `arith add rcx`, `load [rcx]`, `store [rcx]`, and 13 trivial `ret`s. |
| **VM #2** — 8 threaded handlers `ht_0..ht_7` at `0x140001200..0x140001270` sharing the 8-entry table at `0x140002100` | 0 entry sites, 8 threaded sites | The cluster-by-table grouping (entry vs threaded). Every `ht_N` is its own dispatcher whose `function_addr` is in `handlers`, so all 8 land in `threaded_sites`. Handlers are bare tail-dispatchers, classified `null`. |
| **VM #3** — `vm_loop_rich` central dispatcher with the 8-entry table at `0x140002180` | 1 entry site | The full classifier mix — branch (`cmp; jne; ret`), call (direct `call imm`), arith (`add rax, rcx`), load through PC (`mov rax, [rdi]` → `load operand`), store, three `ret`s. |
| **VM #4** — `vm_loop_rip` with `call $+5; pop rsi` RIP-capture, dispatching through the 8-entry table at `0x140002200` (which points at the same handler bodies as VM #3) | 1 entry site | The VMProtect-style `call $+5; pop reg` lookahead at `vm_detect.cpp:704-743`. Run end-to-end:

```sh
python3 tests/fixtures/make_vm_loop_pe.py /tmp/vm_loop.exe
build/cli/ember --vm-detect /tmp/vm_loop.exe
diff -u tests/golden/vm_loop_pe.vm_detect.txt <(build/cli/ember --vm-detect /tmp/vm_loop.exe)
```

The full `ctest` run checks the same goldens — drift means either an
intentional behaviour change (update the golden in the same commit) or
a regression worth digging into.
